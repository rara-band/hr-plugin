# Agent HR — 동작 설계 (v1)

> **한 가지만 판단한다: 현재 에이전트 배치가 방향에 얼마나 핏한가, 그리고 어떻게 더 핏하게 만드나.** 그 외 인사 기능은 v1 범위 밖.
>
> `REQUIREMENTS.md`가 *왜/무엇*이라면 이 문서는 *어떻게*다. 순서: **① 목표 → ② 필요한 맥락 → ③ 수단 → ④ 판정 기준.** doctrine·계약은 [AGENTS.md](AGENTS.md), 개념 매핑은 [docs/concepts-and-mapping.md](docs/concepts-and-mapping.md).

핵심 비유: **실제 HR도 어깨너머로 안 본다.** 근태기록 + 주기적 평가로 일한다. v1은 그 최소 형태다.

---

## ① 목표 — 배치 핏 하나

현재 로스터가 방향에 맞는 배치인지 보고, **더 핏하게 만드는 발령을 권고**한다.

- 발령: **자르기 · 유지 · 나누기**, 그리고 **확실한 경우 채용**. (합치기·구조는 범위 밖 → 나중)
- 원칙: 권고만(자동 반영 없음, 승인 게이트) · 잴 수 있는 신호로만 · 방향은 사람 입력 · 독립성 핵심 역은 안 건드림(직무 분리).

## ② 필요한 맥락 — 딱 셋

| 맥락 | 무엇 |
|---|---|
| **로스터** | `agents/*.md` — 정의된 역 |
| **방향** | 사람이 입력 (점검할 때 물음) |
| **업무량** | worklog — 누가·얼마나 (교대 수) |

**못 재는 것** → 판정 근거로 안 쓰고 한계로 명시: 결과(쓰였나/버려졌나) · 품질(eval) · 비용(토큰) · 메인이 직접 한 일.

## ③ 수단 — 핵심 셋

- **훅 (근태)** — `PostToolUse`(matcher `Task`) = 서브에이전트 한 교대가 끝나 결과를 돌려준 순간 → `~/.claude/plugins/data/hr-plugin/<프로젝트>/worklog.jsonl`에 **한 줄 append**. **레포 밖, 네 코드 안 건드림.**
  - 한 줄 = 한 교대. **페이로드에서 바로 나오는 것(실측 확인):** `role`=`tool_input.subagent_type` · `task`=`tool_input.description` · `session`=`session_id` · 프로젝트=`cwd`. `ts`는 스크립트가 찍는다.
  - `turns`(정밀 업무량)는 페이로드에 없음 → best-effort(생략 가능). **교대 *횟수*(줄 수)가 주 신호.**
  - *`SubagentStop`이 아니라 `PostToolUse(Task)`를 쓰는 이유: 전자는 페이로드에 `role`이 없어 트랜스크립트를 까야 하고, 후자는 `subagent_type`이 입력에 그대로 들어온다.*
  - **"안 불린 역"은 worklog에 없다** → 리뷰 때 `agents/*.md` 명단 ∖ worklog에 나온 역들로 계산. worklog는 *일어난 교대만* 적는 출근부.
  - **보존/정리** — 판단 후 지우지 않는다(되돌아보기에 history 필요). 대신 **오래된 교대만** 덜어낸다: 최근 **90일** + 안전 상한 **2만 줄**(`prune-worklog.sh`), **매 SessionStart에 자동**. 나이 기준이라 되돌아보기를 안 깬다. *보존 기간 90일 ⊇ 되돌아보기 기간(분석가 판단)* — 쓸 수 있는 history는 넉넉히 두고, 얼마나 거슬러 볼진 판단.
- **스킬 `/hr-review` (절차)** — 방향 묻고 → worklog+명단 읽어 `hr-analyst`에 넘기고 → 메모 받아 승인 게이트로. **1터치, 중간/끝 아무 때나.**
- **서브에이전트 `hr-analyst` (판단)** — 아래 ④ 기준으로 판정해 권고 산출. 자기 판단 자기가 승인 안 함.

## ④ 판정 기준 — 배치 핏 2-box

두 축으로 본다. **절대 안 섞는다.**

- **방향 적합** — 이 역이 방향에 닿나? → *사람이 준 기준*
- **업무량** — 이 역이 실제로 불려 일하나? → *측정 신호*

| | 방향에 맞음 | 방향에 안 맞음 |
|---|---|---|
| **일함** | ✅ 유지 (몰리면 나누기) | ⚠️ 방향 재확인 → 정말 불필요하면 자르기 |
| **안 함** (유령) | 🔧 직무 정리 (*자르기 아님* — 필요한데 안 불림) | ✂️ 자르기 |

**"남아도나"는 이렇게 확인한다** (위 표의 *안 함* 행):

1. **안 불렸나** — 되돌아보기 기간에 이 역의 교대 수가 0(또는 미미).
2. **필요했을 국면이 있었나** (제설차 가드) — 방향에 이 역의 일이 있었나 / 그 종류 일이 그동안 실제로 있었나.
3. **판정**:
   - 안 불림 + 방향에도 없음 → 확실히 남아돔 = **자르기 확정**.
   - 안 불림 + 방향엔 필요 → 남는 게 아니라 *빠진 것* = 왜 안 쓰였나 → **개편안(직무 정리)**.
   - 필요한 국면 자체가 없었음(한가한 시즌) → **보류**.

> v1의 "남아돔"은 **안 쓰여서** 남는 것만 본다. 같은 일을 둘이 해서 남는(중복)은 합치기 영역 → 나중.

발령은 **자동 임계값이 아니다**: 신호가 *기울이고* · 가드가 *막고*(직무 분리 · 되돌아보기 기간 짧으면 보류 · 품질로는 안 자름) · 사람이 *승인*한다.

각 발령의 형태:
> 근거(어느 기록 인용) + 로스터 연산(자르기·나누기) + **기대(배치가 얼마나 더 핏해지나)** + 변경 등급 + 한계.

**예시** — 방향 "브랜드 전략 보고서 산출", 최근 20런:

| 역 | 교대 | 판정 |
|---|---|---|
| market-researcher | 12 | ✅ 유지 |
| brand-auditor | 0 | ✂️ 자르기 후보 (유령 + 방향 직접 안 닿음). *가드: 윈도우 짧으면 1회 더 관찰* |
| strategy-planner | 9, 몰림 | ➗ 나누기 (과부하) |

## ⑤ 출력 — 확정 권고 + 개편안 (2단)

사람에게 한 덩어리 리스트를 던지지 않는다. **확신도로 나눈다:**

1. **확정 권고** — 신호가 분명한 것만 *단언조로*. (교대 0 + 방향 무관 = 명백한 자르기 / 방향에 꼭 필요한데 주인 없는 책임 + 기존 역 과부하 = 명백한 채용.) 옵션이 아니다 — 그래도 승인은 거친다. 각 항목: 근거 + 변경 등급 + 한계.
2. **개편안 (시나리오)** — 판단이 갈리는 재배치는 *일관된 묶음 2~3개*로. 각 안에 tradeoff와 "배치가 얼마나 더 핏해지나"를 붙인다. 사람이 하나 고른다. (예: 안A 보수적 = 나누기만 / 안B 적극적 = 나누기 + 직무 정리.)

**가르는 기준 = 확신도.** 한 축이 극단이고 가드 안 걸리면 확정, 손질 방식이 여러 갈래거나 서로 영향 주면 개편안.

## 범위 밖 (나중에)

배치 핏 판정에 *당장* 필요 없는 것은 v1에서 뺀다:

- 결과 캡처(쓰였나/버려졌나) · 합치기 · 관계/구조 판정 · 착수 2터치+브리핑 저장 · Codex 훅 배선 · 시계열 스냅샷 · 품질(eval) · 비용.
- *채용은 "확실한 경우"만 v1 — 빈 책임이 애매하면 개편안으로, 본격 빈틈 분석(직무기술서 대조)은 나중.*

## 아직 열린 것

- `turns`(정밀 업무량)는 `PostToolUse` 페이로드에 없어 best-effort — 필요해지면 트랜스크립트 파싱으로 보강.
- **되돌아보기 기간** — "이 역이 유령이다"라고 하려면 얼마나 과거까지 거슬러 봐야 하나. 고정 숫자(며칠·몇 런)가 아니라, **그 역이 쓸모 있었을 법한 국면이 그 안에 있었는데도 한 번도 안 불렸나**로 본다. 그냥 한가한 시즌이라 0번이면 자르지 말고 보류.
